**princípios / requisitos**

1. Zettels em formato markdown, acessíveis via Finder / Explorer e editáveis em qualquer markdown editor. Prioridade um é: o zettelkasten é app free (agnóstico).
2. Sistema sem custo (open source ou freeware).
3. Sincronia via nuvem para a pasta contendo os zettels.
4. Acessível no celular para consultas, pelo menos.
5. Operável em qualquer sistema operacional simultaneamente.
6. Na pasta principal apenas zettels, anexos em pasta subordinada. 
7. Uma pasta única para todos os zettels comuns. Zettels em Subpastas apenas se integram um projeto (cada projeto uma pasta; mas, importante, cada zettel que pertence a um projeto deve ter tag referente a ele).
8. Nomeclatura dos zettels e anexos sem acentos nem espaços em branco, para compatibilidade do Mac. 
9. Preferencialmente o zettel que não for index deve ter identificador de data, no mínimo (por exemplo 260211), começando o nome do file. Desnecessário haver UIUI. Se for necessário referir um zettel em material externo, criar a UIUI no corpo do zettel. 
10. Os nomes de files poder ter marcadores de classe, como `_J_` para Journal ou `_W_` para Workstuff. A planejar. 
11. Sem preconceito quanto ao conteúdo das notas, mas as tags devem permitir separar o que é texto bruto copiado do que é resumo / coleta. Preferencialmente as cópias de fontes ou as fontes propriamente ditas (íntegras) não deveriam ser zettels, mas anexos.
12. Sem complicação na nomenclatura e na arborização hierárquica: não precisa haver sub-índices ou portais para tudo; mas deve haver temas recorrentes bem definidos, e cada um deles preferencialmente terá um zettel-indexador, que na prática será um zettel-para-aquela tag. Em todo caso o tagueamento é mais relevante que a indexação. 
13. Em princípio os anexos são de todo tipo, imagens, PDFs, zips e docs. O zettel deve referir ou linkar seus anexos.
14. O nome do anexo deve permitir fácil reconhecimento do zettel a que é anexo.
15. Sem necessidade de importar todo o passado.
16. Notas provisórias ou temporárias devem conter tag de apagar com data.
17. Desnecessário haver pasta separada para entradas bibliográficas, índice onomástico, journals, tags/categorias/classes. As tags dão conta de tudo. 
18. Opcional mas MUITO desejável: links funcionais, ainda que via macro ou automação powershell.
19. Opcional mas MUITO desejável: um app que corrige os links internos quando o nome do zettel linkado for alterado.
20. Opcional mas muito desejável: achar uma IA capaz de conversar com os zettels.
21. Opcional mas desejável: compatibilidade com programa gerenciador tipo Obsidian, que preserve a independência e editabilidade externa dos zettels. 
22. Opcional mas desejável: função autocompletar, especialmente para links internos e tags.
23. Opcional mas desejável: zettelkasten gerencia takslist, contas a pagar, contas pagas. 
24. Opcional mas desejável: exportação de notas para html e PDF.
25. Opcional mas desejável: geração de TOC navegável.
26. Opcional mas desejável: congregação de zettels. Significa poder escrever os capítulos em files separados, e usar um file principal com os links/títulos dos zettels-capítulos e ao final exportar o texto integral do projeto num html/PDF só, usando o zettel-index como gerenciador para ordenar os capítulos.
27. Opcional mas desejável: aparência altamente personalizável.
28. Opcional mas desejável: preview em html.
29. Opcional mas desejável: mouse-free.
